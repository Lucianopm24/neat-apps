OAuth de Neat & Sistema de Tenants (Neat ID)

Documentación técnica del flujo de autenticación de Neat: el OAuth de la cuenta global de Neat, y el sistema multi-tenant Neat ID que permite a cualquier desarrollador crear su propia app con login propio (email/password, "Continuar con Neat" o login social) sin escribir un backend de autenticación.

OAuth 2.0 + PKCE Multi-tenant OIDC-compatible JWT (HS256)
ℹ️

Audiencia: este documento está pensado para quien mantiene la infraestructura de Neat y para desarrolladores que integran un Tenant (una app propia) contra Neat ID. Se basa directamente en el código fuente del backend (neat-apps-b.vercel.app) y en los tres frontends: oauth.html, index.html y admin.html.

Arquitectura general

Neat tiene en realidad dos sistemas de autenticación apilados:

1. Neat OAuth (cuenta global)

El sistema de cuentas "de toda la vida" de Neat (users). Es la identidad raíz: la que usa Neat Forums, Neat Chatter, Neat Points, etc. Funciona como un proveedor OAuth 2.0 clásico con scopes propios del ecosistema Neat.

2. Neat ID (Tenants)

Una capa multi-tenant construida encima del OAuth global. Cualquier usuario de Neat puede crear un Tenant (una "App" en admin.html) que obtiene su propio pool de usuarios (id_app_users), su propia pantalla de login hosteada, y puede aceptar login local, login social, o "Continuar con Neat".

La pieza que conecta ambos sistemas es un client OAuth interno (PKCE, público) que Neat genera automáticamente para cada Tenant al crearlo. Ese client interno es el que usa la hosted login page para pedirle a un usuario "¿confirmas que quieres entrar a Tenant X con tu cuenta Neat?" — exactamente la misma pantalla de oauth.html que usaría cualquier app de terceros.

Vista de alto nivel: el Tenant nunca ve contraseñas de Neat. Solo recibe un code y lo cambia por un token de su propio sistema (Neat ID), no por un token de la cuenta global.

Dominios y responsabilidades

DominioQué sirveRol en el flujo

Neat OAuth — Cómo funciona

Es un proveedor OAuth 2.0 + PKCE estándar sobre la cuenta global de Neat (colección users). Cualquier app externa (o el propio sistema Neat ID, como veremos después) puede registrar un client y pedirle a un usuario de Neat que autorice acceso a ciertos scopes de su cuenta.

  • Soporta flujo público (PKCE) — sin client_secret, usando code_challenge/code_verifier — y flujo confidencial clásico con client_secret.
  • El code de autorización expira a los 5 minutos y es de un solo uso.
  • El access_token resultante es un JWT firmado (HS256). Su duración depende del scope: 24 horas para tokens normales, 30 días para tokens con scope account (acceso completo).
  • Si se pide el scope openid, además del access_token se emite un id_token (JWT con claims sub, username, email, verified).

Los scopes sensibles llegan desmarcados por defecto en oauth.html aunque la app los haya pedido — el usuario debe activarlos manualmente, y si el client no está verificado se le pide confirmación extra. El scope account siempre muestra una advertencia explícita ("equivale a tu contraseña") antes de poder activarse.

Flujo paso a paso (Authorization Code + PKCE)

  1. 1
    Redirect inicialLa app de terceros redirige al usuario a https://neat.blue/oauth.html con query params: client_id, redirect_uri, scope, state, code_challenge, code_challenge_method.
  2. 2
    Carga de la info del clientoauth.html llama a GET /oauth/clients/:clientId (público) para mostrar nombre, badge de "app verificada" y armar la lista de scopes pedidos.
  3. 3
    Selección de cuenta / loginEl usuario elige una cuenta guardada en localStorage (neat_oauth_accounts) o inicia sesión con POST /chat/login. Las cuentas se guardan localmente junto con su propio token de sesión real (no OAuth).
  4. 4
    Ajuste de scopesEl usuario activa/desactiva los scopes sensibles. Si activa account, ve la advertencia de "acceso completo".
  5. 5
    AutorizarPOST /oauth/authorize (con el token de sesión real del usuario en el header) crea un registro en oauth_codes y devuelve un code de un solo uso, válido 5 minutos.
  6. 6
    Redirect de vueltaoauth.html redirige el navegador a redirect_uri?code=...&state=....
  7. 7
    Intercambio del codeEl backend de la app de terceros llama a POST /oauth/token con el code y, según el tipo de client, code_verifier (PKCE) o client_secret (confidencial). Recibe access_token (y id_token si pidió openid).
  8. 8
    Consumo del tokenLa app usa el access_token como Authorization: Bearer ... contra GET /oauth/userinfo o GET /oauth/kv, que filtran la respuesta según los scopes concedidos.

Flujo Authorization Code + PKCE de Neat OAuth (cuenta global)

Ejemplo — armar la URL de autorización

JavaScript (lado cliente) 
// 1. Generar el par PKCE
const verifier = randomString(64);
const challenge = base64url(await crypto.subtle.digest("SHA-256", encode(verifier)));

// 2. Redirigir al usuario
const params = new URLSearchParams({
  client_id: "<tu_client_id>",
  redirect_uri: "https://tu-app.com/callback",
  scope: "openid profile email",
  code_challenge: challenge,
  code_challenge_method: "S256",
  state: crypto.randomUUID()
});
location.href = `https://neat.blue/oauth.html?${params}`;
Intercambio del code (backend) 
const res = await fetch("https://neat-apps-b.vercel.app/oauth/token", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    code: req.query.code,
    clientId: "<tu_client_id>",
    redirectUri: "https://tu-app.com/callback",
    codeVerifier: verifierGuardadoEnSesion
  })
});
const { access_token, id_token, expires_in, scopes } = await res.json();

Neat ID — Qué es un Tenant

Un Tenant (llamado internamente id_app en la base de datos, y simplemente "App" en el panel admin.html) es un espacio de autenticación independiente: tiene su propio pool de usuarios (id_app_users), su propia URL de login hosteada, su propio branding, y puede tener múltiples clients OAuth (web, móvil, etc.).

🏷️

Cada Tenant se identifica por un slug único (ej. innernet). Ese slug determina su URL de login: id.neat.qzz.io/innernet, y su panel de cuenta de usuario final: id.neat.qzz.io/innernet/manage.

Cuando se crea un Tenant, el backend genera automáticamente dos cosas además de la propia App:

  • Un client interno (PKCE, público, sin secret) en la colección oauth_clients del sistema de Neat OAuth global — este es el que permite el botón "Continuar con Neat". Sus redirectUris están fijas: neat-apps-b.vercel.app/id/callback y id.neat.qzz.io/<slug>/neat-callback.
  • Un client "Default" en id_app_clients — confidencial (con clientSecret), usado por el login local/registro hasta que el dev cree clients propios.

Crear un Tenant (panel admin.html)

admin.html es el panel de administración general. El dueño de un Tenant entra con su cuenta global de Neat (POST /chat/login — mismo sistema que el resto de Neat, no el de Tenants) y desde ahí gestiona sus Apps.

POST /id/apps

Crea un nuevo Tenant. Límite: 1 app gratis por usuario, ilimitadas con Neat Plus.

Body 
{
  "name": "InnerNet",
  "description": "Red social interna del equipo",
  "redirectUris": ["https://innernet.app/callback"],
  "logoUrl": "https://innernet.app/logo.png",
  "primaryColor": "#6366f1",
  "homepageUrl": "https://innernet.app"
}
Respuesta 
{
  "slug": "innernet",
  "name": "InnerNet",
  "branding": { "logoUrl": ..., "primaryColor": "#6366f1" },
  "defaultClient": { "clientId": "...", "clientSecret": "...", "isPublic": false },
  "internalClientId": "...",
  "appKey": "...",
  "loginUrl": "https://id.neat.qzz.io/?app=innernet",
  "createdAt": "..."
}
🔒

clientSecret solo se devuelve una vez, en el momento de creación (o al rotar el secret). No se puede volver a consultar después. appKey en cambio es pública y siempre recuperable — es la credencial usada por apps sin backend para leer/escribir su propio KV storage.

El slug se genera automáticamente a partir del name y nunca puede coincidir con una ruta reservada del frontend (ej. admin), para no romper el ruteo de id.neat.qzz.io.

Login local (email / password)

Esta es la pantalla principal de index.html cuando se accede a id.neat.qzz.io/<slug>?redirect_uri=.... El flujo entero ocurre dentro de la hosted login page — el code que produce ya es del Tenant, no de Neat global.

  1. 1
    Boot de la páginaAl cargar, index.html detecta el slug por el path (/innernet) y llama en paralelo a GET /id/apps/:slug/public (branding, nombre) y GET /id/apps/:slug/page-token (un JWT corto de 15 min que autoriza login/registro en esta carga de página).
  2. 2
    Registro (si es nuevo)POST /id/users/:slug/register con { email, password, username, internalToken }. Si el Tenant tiene los registros cerrados, requiere además un inviteToken válido.
  3. 3
    LoginPOST /id/users/:slug/login con { email, password, internalToken, redirectUri, state, codeChallenge, codeChallengeMethod, clientId }. Si el usuario tiene 2FA, responde { totp_required: true, totp_pending_token } en vez del code.
  4. 4
    Code del TenantEl backend genera un code (tipo id_app_local) en oauth_codes, asociado al idAppUserId (no a un username de Neat global), y devuelve { code, redirectUri, state, pageSessionToken }.
  5. 5
    Oferta de vincular NeatAntes de redirigir, index.html muestra una pantalla opcional ("link-neat-offer") ofreciendo vincular la cuenta recién logueada con la cuenta global de Neat. El usuario puede aceptar o continuar directo.
  6. 6
    Redirect finalEl navegador va a redirectUri?code=...&state=... — la URL del Tenant, registrada en su client.
💡

El internalToken (de page-token) es distinto del code OAuth: es solo un JWT de corta duración que prueba que el login/registro viene realmente de la hosted page de ese slug, no de una request falsificada directa al backend.

"Continuar con Neat" (login con la cuenta global)

Este es el punto donde ambos sistemas se conectan. Desde la hosted login page, el botón "Continuar con Neat" no llama a la API del Tenant — redirige a oauth.html (la pantalla de consentimiento de Neat global) usando el client interno PKCE que se creó junto con el Tenant.

"Continuar con Neat": el usuario nunca ve credenciales del Tenant, solo aprueba en la pantalla de Neat global.

  1. 1
    Armar el stateEl frontend codifica en base64url un JSON con todo lo necesario para terminar el login después: { appSlug, clientId, redirectUri, codeChallenge, codeChallengeMethod, tenantState, codeVerifier }.
  2. 2
    Redirigir a oauth.htmlCon client_id = internalClientId del Tenant y, crucialmente, redirect_uri = https://neat-apps-b.vercel.app/id/callback (el backend, no la hosted page).
  3. 3
    Usuario aprueba en Neat globalMisma pantalla de oauth.html descrita arriba, con scope openid profile email.
  4. 4
    GET /id/callbackNeat redirige aquí con code + state. El backend decodifica el state, intercambia el code Neat por un access_token Neat (llamando a su propio /oauth/token), y obtiene el perfil con /oauth/userinfo.
  5. 5
    Resolver / crear id_app_userBusca un usuario del Tenant ya vinculado a ese neatUserId. Si no existe pero hay un usuario local con email @neat.blue coincidente, lo vincula. Si no existe ninguno, crea uno nuevo (salvo que el Tenant tenga registros cerrados).
  6. 6
    Code del Tenant + redirectEmite un code (tipo id_app_neat) y redirige directo al redirectUri del Tenant — el navegador nunca vuelve a pasar por la hosted login page en este caso.
⚠️

Regla de sincronización importante: si el id_app_user ya tiene passwordHash (cuenta local con contraseña propia), su email/username nunca se sobrescriben con los de Neat — solo se actualiza lastNeatSync. Solo se sincronizan esos campos para usuarios 100% Neat (sin password local), porque no tienen una identidad local que proteger.

Vincular cuenta Neat desde una sesión local activa

Distinto del caso anterior: aquí el usuario ya inició sesión localmente (email/password) en el Tenant, y quiere además vincular su cuenta Neat — para poder entrar después con cualquiera de los dos métodos. El retorno, en este caso, sí vuelve a la hosted page (no al backend), en una ruta dedicada: id.neat.qzz.io/<slug>/neat-callback.

  1. 1
    Guardar estado en sessionStorageAntes de salir, la hosted page guarda neat_link_verifier, neat_link_page_token (el pageSessionToken del login local) y los datos del code local pendiente.
  2. 2
    Redirect a oauth.htmlCon redirect_uri = id.neat.qzz.io/<slug>/neat-callback (vuelve a la propia página, no al backend).
  3. 3
    POST /id/users/:slug/link-neatBody: { neatCode, localAccessToken, redirectUri, codeVerifier }. localAccessToken acepta tres tipos de token: id_app (sesión real), id_app_page_session (token corto de 10 min del login recién hecho) o manage_session (desde el panel de cuenta).
  4. 4
    Validación de unicidadEl backend verifica que esa cuenta Neat no esté ya vinculada a otro usuario de este mismo Tenant antes de guardar neatUserId.
  5. 5
    Continuar el login originalUna vez vinculado, la página retoma el code local que ya tenía pendiente y redirige al Tenant normalmente.

Para desvincular: DELETE /id/users/:slug/link-neat (requiere token id_app o manage_session). Solo se permite si el usuario tiene contraseña local — si no, perdería todo acceso a su cuenta.

Login social (Google, GitHub, etc.)

Cada Tenant puede habilitar proveedores OIDC de terceros, de dos formas: proveedores globales (configurados una vez por Neat y reutilizables por cualquier Tenant) o proveedores propios del Tenant (su propio client_id/discovery URL). Ambos se gestionan desde la pestaña "Social" de admin.html.

MétodoRutaDescripción
POST/id/social/globalCrear un proveedor global (solo backend/admin de Neat).
POST/id/apps/:slug/socialCrear un proveedor propio del Tenant.
GET/id/apps/:slug/social/availablePública — lista los proveedores activos para mostrar botones en la hosted page.
GET/id/users/:slug/social/:key/startInicia el flujo: redirige al authorizationEndpoint del proveedor (PKCE generado en el backend).
GET/id/users/:slug/social/:key/callbackEl proveedor redirige aquí; el backend canjea el code, obtiene el perfil y resuelve/crea el id_app_user.
POST/id/users/:slug/social/link-existingConfirma con password local y vincula el proveedor social a una cuenta existente con el mismo email.

El patrón es el mismo que "Continuar con Neat", pero genérico para cualquier proveedor OIDC:

  1. 1
    StartEl backend genera un flowId + par PKCE, lo guarda en social_login_flows junto con los datos OAuth del Tenant (client, redirectUri, state), y redirige al proveedor externo.
  2. 2
    Callback del proveedorRecibe code + state (el flowId). Canjea el code en el tokenEndpoint del proveedor y llama a su userinfoEndpoint.
  3. 3
    Resolución de cuentaBusca por social_<key>_sub. Si no existe pero el email coincide con una cuenta con password local, ofrece vincular (link_token, pantalla "link-social" en index.html). Si no hay coincidencia, crea un id_app_user nuevo con emailVerified: true (entrar por OAuth social ya es prueba de identidad).
  4. 4
    Code y redirectEmite un code (tipo id_app_social) y redirige directo al Tenant — igual que con Neat global, no vuelve a pasar por la hosted page.

Panel /manage — cuenta del usuario final

Ruta: id.neat.qzz.io/<slug>/manage. Es deliberadamente independiente del flujo OAuth: no usa client_id, client_secret, code ni PKCE. El usuario entra directo con su email/contraseña de ese Tenant para gestionar su propia cuenta (cambiar password, ver sesiones activas, activar 2FA, vincular/desvincular Neat).

🔑

El token que emite este panel tiene type: "manage_session" — distinto de type: "id_app" (el de OAuth normal). Esto es intencional: un token de manage nunca debe servir en /id/userinfo, /id/token ni en el KV storage, y viceversa. Vida corta: 30 minutos.

Intercambio de code → token (/id/token)

Equivalente a /oauth/token pero para el ecosistema de Tenants. Lo llama el backend del desarrollador (o el frontend, si su client es público/PKCE) para canjear cualquier code de tipo id_app_local, id_app_neat o id_app_social por un access_token real.

Intercambio de code por token — válido sin importar si el login original fue local, con Neat o social.

Request 
const res = await fetch("https://neat-apps-b.vercel.app/id/token", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    code: "<code recibido en tu redirect_uri>",
    client_id: "<tu clientId>",
    client_secret: "<tu clientSecret>",  // omitir si tu client es público (PKCE)
    redirect_uri: "https://innernet.app/callback",
    code_verifier: "<solo si usaste PKCE>"
  })
});
const { access_token, id_token, expires_in, scope } = await res.json();
access_token (payload del JWT) 
{
  "type": "id_app",
  "appSlug": "innernet",
  "sub": "<id_app_user._id>",
  "username": "...",
  "email": "...",
  "neatUserId": "<o null>",
  "isNeatUser": true | false,
  "groups": { "moderadores": ["admin"] }
}

Cada intercambio exitoso guarda además una sesión en id_app_sessions (visible y revocable desde /manage/sessions) y el token expira a las 24 horas.

Userinfo del Tenant

GET /id/userinfo

Con Authorization: Bearer <access_token> (debe ser tipo id_app). Devuelve { sub, username, email, appSlug, isNeatUser, neatUserId, verified, createdAt, groups }.

KV Storage

Pensado para Tenants sin backend propio — apps puramente frontend que igual necesitan persistir datos. Dos espacios separados:

KV privado por usuario

Un blob JSON por usuario. Para escribir el dato de un usuario se necesita siempre su propio access_token — ni siquiera la appKey pública alcanza por sí sola (modelo equivalente a "anon key" + RLS de Supabase). Límite configurable, 25KB por defecto, 1MB como techo absoluto.

KV público por Tenant

Un solo blob compartido por todo el Tenant (config global, leaderboard, etc.). Lectura sin auth para cualquiera; escritura exclusiva del backend del dev vía x-client-id + x-client-secret — la appKey nunca puede escribirlo, para que ningún visitante corrompa el dato compartido.

Discovery OIDC

Neat ID expone discovery global y discovery por Tenant (path-based, ya que un solo dominio sirve N pools de usuarios distintos — mismo patrón multi-tenant que WorkOS o Clerk):

RutaAlcance
GET /id/.well-known/openid-configurationConfiguración OIDC global del sistema Neat ID.

GET /id/.well-known/jwks.jsonJWKS — actualmente vacío porque los tokens usan HS256 (firma simétrica), no RS256.

Notas de seguridad

  • PKCE obligatorio en clients públicos: si un client es isPublic: true, el backend rechaza el intercambio si el code original no tuvo code_challenge — un client público nunca puede usar client_secret como alternativa.
  • Validación estricta de redirect_uri: en cada paso (autorización, callback de Neat, callback social, intercambio de token) se compara contra la lista exacta registrada en el client correspondiente. No hay coincidencia parcial ni wildcards.
  • Separación de tipos de token: oauth (Neat global), id_app (Tenant), id_app_page_session (puente de 10 min), manage_session (panel de cuenta), totp_pending (2FA pendiente). Cada middleware valida explícitamente el type esperado — un token de un contexto nunca es válido en otro, aunque esté correctamente firmado.
  • Aislamiento entre Tenants: todas las consultas a id_app_users, id_app_clients, id_app_memberships, etc. están filtradas por appSlug — un usuario del Tenant A nunca es visible ni accesible desde el Tenant B, aunque comparta la misma cuenta Neat global vinculada.
  • Vinculación 1-a-1 con Neat global: un mismo neatUserId no puede vincularse dos veces dentro del mismo Tenant — pero sí puede estar vinculado en N Tenants distintos (cada uno con su propio id_app_user independiente).
  • requireAuth bloquea tokens OAuth: endpoints de gestión sensible (crear/borrar Tenants, rotar secrets, cambiar password) usan requireAuth, que rechaza explícitamente cualquier token type: "oauth" — solo sirven con una sesión real de Neat (login directo), nunca con un token obtenido por una app de terceros.
